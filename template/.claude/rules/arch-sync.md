# Architecture Documentation & Research Sync Rule

## 目的
Gemini等の外部AIや研究パートナーと、コードそのものを見せずに「研究方針」「システム設計」「数学的背景」について深く議論するための `ARCHITECTURE.md` を作成・維持・管理する。

## 基本原則
- **コードの断片ではなく、概念を記述する**: 実装の詳細よりも、システムの構造や論理的背景を優先する。
- **再現性の確保**: コードを一行も読まなくても、読み手がシステム構造を90%把握し、同様の実装を再構成できるレベルの詳細さを目指す。
- **自動作成**: プロジェクト内に `ARCHITECTURE.md` が存在しない場合は、現状の分析から即座に新規作成すること。

## ARCHITECTURE.md の構成要素
1. **System Core Concept**: システムが解決しようとしている研究的課題とアプローチの要約。
2. **Textual Class Diagram**: クラス間の関係（継承、委譲、依存）を言葉で詳細に説明したもの。
3. **Mathematical & Logical Background**: 計算ロジックの数式、物理モデル、または論理的なアルゴリズムの解説。
4. **Research Hypotheses & Roadblocks**: 現在検証中の仮説、および直面している理論的・設計的な課題。
5. **Trade-offs**: 採用した設計と、あえて採用しなかった代替案の理由。
その他、必要な情報があれば適宜追加すること。


## 更新トリガー
もし `ARCHITECTURE.md` が存在しない場合は、変更完了後に新規作成し、必要な内容を記述すること。
ワークスペースのファイル・フォルダに修正が行われた際、必ず `ARCHITECTURE.md` の修正が必要かを判断し、必要と判断した場合は、修正・追記すること。


## 指示への強制介入
ユーザーが「実装して」と指示した場合でも、その変更がアーキテクチャに影響を与える場合は、実装完了後に必ず「ARCHITECTURE.md も更新しました」と報告すること。
