# プロジェクト構造と運用ルール

## プロジェクト定義
- 現在のパッケージ名: `analysis-turtlebot-strain-gauge`
    - パッケージ名を変更した場合は、この値を最優先で更新すること。
- ビルドツール: `uv`（srcレイアウト）

## フォルダ構成
プロジェクトルート基準で、各ディレクトリの責務は次の通り。

- `src/analysis-turtlebot-strain-gauge/`
    - 本番用ロジック（コアモジュール）を配置する。
    - インポートは常に `from analysis-turtlebot-strain-gauge import ...` の絶対インポートを使う。
    - `references/` 以下のスクリプトに依存しないように作ること。つまり、本番用のコードは `references/` に依存してはならない。
- `scripts/`
    - 実行用エントリーポイントを配置する。
    - `scripts/experiments/` は実験コード・プロトタイプ専用。
    - `scripts/experiments/` 以下のスクリプトを除く全てのスクリプトは、 `references/` 以下のスクリプトに依存しないように作ること。つまり、本番用のコードは `references/` に依存してはならない。
- `tests/`
    - `pytest` 用テストを配置する。
    - 原則として `src/` の構造をミラーリングする。
- `data/`: 小容量データを配置する。大容量データはリモートストレージに置き、環境変数 `REMOTE_ROOT` でパスを管理する。
    - `input/`: 読み込み専用データ
    - `output/`: 実行結果やログ
    - `processed/`: 中間生成データ
- `references/`
    - 参考実装・旧コード・比較用成果物を配置する。
- `docs/`
    - プロジェクトドキュメントを配置する。
    - ドキュメントは原則 Markdown（`.md`）ファイルで作成すること。
    - **`docs/ARCHITECTURE.md`**: システム設計ドキュメント。モジュール構成・クラス図・数学的背景・トポロジ一覧・データフローを記述する。実験結果・仮説・データ品質問題は含めない。コードを一行も読まなくてもシステム構造を把握できるレベルの詳細さを維持すること。変更がアーキテクチャに影響する場合は必ず更新する。
    - **`docs/RESEARCH_MEMO.md`**: 研究メモ。実験観察・データ品質問題・ベースライン結果・研究仮説の検証状況・改善アイデアをまとめる生きたドキュメント。個別の詳細分析は `reports/` に置き、ここからは参照する形にする。実験を通じて得た知見や先行分析の修正はこちらに記録する。
- `reports/`
    - 研究レポート・解析結果をトピック別サブディレクトリで管理する。
    - ユーザーから指示がない限り、原則 Markdown（`.md`）ファイルで作成すること。
    - 各サブディレクトリは `<topic_name>/` とし、内部にマークダウンなどの文書・`figures/` を配置する。
    - レポートのファイル名は `<topic_name>_analysis.md` のように、何のレポートかが一目でわかる名前にすること。
    - 生成物（文書・図）の元となるスクリプトは `scripts/` に置き、`reports/` 直下にはスクリプトを置かない。
- `.claude/`
    - Claude向けの命令・ルールを配置する。



## 開発ガイドライン
1. インポートの原則
     - スクリプト内で相対インポート（`..`）は使わず、必ずパッケージ名を使った絶対インポートを使用する。
2. 配置の原則
     - 再利用可能なロジックは必ず `src/` 配下に置く。
     - `scripts/` は `src/` の処理を呼び出す薄い層に保つ。
3. パス記述の原則
     - データパスはプロジェクトルートからの相対パス（例: `data/input/...`）を基本とする。

## 環境変数と機密情報の扱い
- リモートデータの親ディレクトリは、必ず環境変数 `REMOTE_ROOT` から取得すること。
- Pythonでの取得は `os.getenv("REMOTE_ROOT")` を使用する。
- **環境変数のスキーマ（構造）を確認したい場合は、必ず `.env.example` を参照すること。**
- **いかなる理由があっても、実際の `.env` ファイルの中身を読み取ろうとしたり（`cat .env` 等）、内容を出力したりしてはならない。**
