以下の手順で git commit を対話的に行ってください。分割が必要な場合は、1コミットごとにこの手順を繰り返します。

---

### Step 0: コミットモードの開始

最初に以下を実行してください。

```
touch /tmp/claude_commit_mode
```

---

### Step 1: 会話履歴からコンテキストを読み取る

現在の会話履歴を振り返り、以下を把握してください。
- **何を・なぜ変更したか**（実装の動機・背景・設計判断）
- **解決した問題や達成した目標**
- **ユーザーが重視していた制約や方針**

この理解をコミットメッセージの "why" に反映させることが最重要です。

---

### Step 2: 変更の全体把握

以下を実行して変更状況を確認してください。

```
git status
git diff HEAD
```

変更ファイルを俯瞰し、**論理的なまとまり**でコミットを分割すべきか判断してください。
- 1つのまとまりで問題ない場合 → Step 3へ
- 分割すべき場合 → 最初のまとまりについて Step 3 を進め、残りは後続サイクルで処理

---

### Step 3: コミット内容の提示

以下の形式でユーザーに提示してください。

````
## コミット案

**ステージするファイル:**
- path/to/file1.py
- path/to/file2.py

**コミットメッセージ:**
```
<type>: <タイトル（50文字以内）>

- 変更点1（背景・理由を含む）
- 変更点2
```

**このコミットの意図:**
（会話の流れから読み取った、このコミットが表現すべき背景を1〜2文で）
````

その後、以下の選択肢を提示してください。

```
承認して次へ進む場合: そのまま Enter または「ok」
メッセージを修正する場合: 修正後のメッセージ全文を貼り付け
ステージ対象を変更する場合: 「〇〇だけにして」など指示
中断する場合: 「やめる」
```

---

### Step 4: ユーザーの応答に従って処理

**承認された場合:**
```
git add <対象ファイル>
git commit -m "$(cat <<'EOF'
<確定したコミットメッセージ>
EOF
)"
```

**メッセージ修正が指示された場合:**
ユーザーが貼り付けた内容をそのままコミットメッセージとして使用し、Step 3 の表示を更新してから再度承認を求めてください。（編集ループは承認されるまで繰り返す）

**ステージ対象の変更が指示された場合:**
ステージするファイルを更新して Step 3 に戻り、再提示してください。

---

### Step 5: 残りの変更を確認

コミット後に `git status` を実行し、未コミットの変更が残っている場合は以下を確認してください。

```
残りの変更があります。続けてコミットしますか？
（続ける場合: 「続ける」 / 終了する場合: 「終わり」）
```

「続ける」と言われた場合は Step 2 から繰り返します。

---

### Step 終了: コミットモードの解除

すべてのコミットが完了した場合、またはユーザーが「やめる」と言って中断した場合は、必ず以下を実行してください。

```
rm -f /tmp/claude_commit_mode
```

---

### ⚠️ 厳守事項

- `.env` や認証情報を含むファイルをステージしないこと。該当ファイルを見つけた場合は必ずユーザーに警告すること。
- コミットメッセージのタイトルは **英語・50文字以内**、本文は箇条書きで簡潔に。
- `git add -A` や `git add .` は使わず、ファイルを個別に指定すること。
- ユーザーが指定したメッセージは一字一句変えずに使うこと。
